全局 IM 系统 — 产品方案设计
阶段二 · 方案设计
一、整体方案思路

入口设计：全局消息入口采用侧边悬浮按钮（右下角固定），始终可见，点击展开会话列表抽屉。入口仅对已购课用户展示，未购课用户完全隐藏。

场景区分：系统通过当前页面上下文自动判断用户所处场景。课中场景（在课程/代码题页面）仅展示当前课程对应班主任的单一会话入口；课外场景（其他所有页面）展示全部班主任的会话列表，支持选择任意班主任发起对话。

通知机制：复用现有 WebSocket 长连接，班主任在后台主动发消息时，通过 WebSocket 实时推送至学生端，在页面顶部弹出气泡通知，悬浮入口未读角标同步更新。

会话聚合：每个班主任对应一个独立会话，会话按最新消息时间倒序排列。一期历史消息以历史记录形式并入对应会话，无需额外迁移。

二、核心设计决策

决策① 侧边悬浮入口 推荐

右下角固定悬浮按钮，不干扰主内容流，未读角标始终可见。相较顶部导航铃铛，触达路径更短；相较底部 Tab，不占用核心导航位。

决策② 场景感知会话列表 推荐

通过路由上下文判断课中/课外，课中仅展示当前班主任避免干扰学习，课外展示完整列表给予充分选择。列表从右滑出抽屉，不跳转页面保持沉浸。

决策③ WebSocket 复用推送 推荐

复用一期已建立的 WebSocket 长连接复用，零额外成本实现实时消息推送。气泡通知定位页面顶部，3秒自动消失，支持点击跳转。

决策④ 未读数上限 99+ 推荐

每个会话独立累计未读数，悬浮入口角标展示所有会话未读总数。上限显示 99，避免数字溢出，兼顾大屏可读性与视觉简洁。

五、后端接口设计（参考）
方法 接口路径 接口说明
GET /api/im/conversations 获取全局会话列表，返回该用户所有班主任的会话信息（含未读数、最新消息）
Query: scene=in_course（课中）/ scene=all（课外），用于过滤当前课程班主任
GET /api/im/conversations/{id}/messages 获取指定会话的消息历史，支持分页
Query: page, size，用于滚动加载历史消息
POST /api/im/conversations/{id}/messages 发送消息
Body: { content: string }，发送成功后触发 WebSocket 推送
POST /api/im/conversations/{id}/read 标记指定会话已读
将指定会话的未读数清零，同步更新悬浮入口角标总数
GET /api/im/unread/total 获取全局未读总数
用于前端初始化时设置悬浮入口角标，避免全量拉取会话列表

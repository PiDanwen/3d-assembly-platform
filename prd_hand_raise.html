需求背景

业务背景

在线编程课平台目前已建立完善的课程学习体系，学生可在平台学习视频课程、完成编程作业。随着业务发展，需要为已购课学生提供便捷的与班主任沟通渠道，提升服务体验和学习效果。

现状与问题

核心痛点
  • 触达效率低：学生遇到问题只能通过课程详情页联系班主任，路径深、转化率低，大量学生放弃求助
  • 消息无感知：班主任回复消息后，学生无法实时收到通知，容易错过重要信息
  • 多课程割裂：购买多门课程的学生需要切换多个页面管理不同班主任的对话，体验碎片化
  • 入口隐蔽：现有消息入口位置不明显，学生遇到问题时找不到客服入口，影响用户满意度

触发此需求的原因

运营团队反馈：学员求助率低于预期，很多学员遇到问题后选择沉默或直接退出课程。班主任团队反馈：主动触达学员时缺乏有效渠道，无法及时传递课程通知和个性化关怀。通过用户调研，学员普遍表示希望有一个"随时能找到老师"的感觉。

📌 关键约束
  • 仅对已购课用户展示，未购课用户完全隐藏
  • 需支持 APP端（iOS/Android）、PC端、Pad端三端
  • 课中场景（学习页面）需特殊处理，避免干扰学习
  • 复用现有 WebSocket 长连接，零额外成本

产品方案

5.1 方案概述

全局消息入口采用侧边悬浮按钮（右下角固定），始终可见，点击展开会话列表抽屉。入口仅对已购课用户展示，未购课用户完全隐藏。

系统通过当前页面上下文自动判断用户所处场景。课中场景（在课程/代码题页面）仅展示当前课程对应班主任的单一会话入口；课外场景（其他所有页面）展示全部班主任的会话列表，支持选择任意班主任发起对话。

复用现有 WebSocket 长连接，班主任在后台主动发消息时，通过 WebSocket 实时推送至学生端，在页面顶部弹出气泡通知，悬浮入口未读角标同步更新。

5.4 关键设计决策

设计决策① 侧边悬浮入口

右下角固定悬浮按钮，不干扰主内容流，未读角标始终可见。相较顶部导航铃铛，触达路径更短；相较底部 Tab，不占用核心导航位。

设计决策② 场景感知会话列表

通过路由上下文判断课中/课外，课中仅展示当前班主任避免干扰学习，课外展示完整列表给予充分选择。列表从右滑出抽屉，不跳转页面保持沉浸。

设计决策③ WebSocket 复用推送

复用一期已建立的 WebSocket 长连接，零额外成本实现实时消息推送。气泡通知定位页面顶部，3秒自动消失，支持点击跳转。

设计决策④ 未读数上限 99+

每个会话独立累计未读数，悬浮入口角标展示所有会话未读总数。上限显示 99，避免数字溢出，兼顾大屏可读性与视觉简洁。

技术方案

系统架构

整体架构
  • 客户端：iOS App、Android App、PC Web、Pad Web
  • 展示层：悬浮入口组件、会话列表组件、聊天详情组件、气泡通知组件
  • 业务层：会话状态管理、WebSocket 连接、消息路由、场景感知逻辑
  • 服务层：消息发送服务、会话查询服务、未读数服务、历史消息服务
  • 数据层：会话表 (im_conversation)、消息表 (im_message)、用户表、班主任-课程关系表

核心接口

方法 接口路径 说明
GET /api/im/conversations 获取全局会话列表（含未读数、最新消息）
Query: scene=in_course/all
GET /api/im/conversations/{id}/messages 获取指定会话的消息历史（分页）
Query: page, size
POST /api/im/conversations/{id}/messages 发送消息
Body: { content: string }
POST /api/im/conversations/{id}/read 标记指定会话已读
GET /api/im/unread/total 获取全局未读总数（初始化角标用）
